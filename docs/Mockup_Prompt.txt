Beacon Project

This application will allow employees to Create/Edit/Remove Advertisers, Campaigns, Ad Groups, and Connectors. The connectors will be a connection configuration to connect with third party vendors like a DSP or Ad Server.

Hierarchy is from top to bottom: Advertiser -> Campaign -> Ad Group
1.	A user will create an Advertiser, from the side panel. (If it does not exist)
2.	Then the user can either create the Campaign or Ad Group next, from the panel.  If the Ad Group was first, there will be an option to create the campaign from the Ad Group fly-in panel.
3.	If the user creates the campaign first, from the panel, then there will be an option at the bottom of the campaign form, to create and assign an ad group to that campaign.  If clicked to add the ad group, the campaign will auto-save.
4.	Connectors will be shown and selected from the Campaign level.


Application built with:
⦁	HTML
⦁	JavaScript
⦁	CSS

File Hiearchy:
Pages
⦁	Advertisers (Landing Page - Shows all of the Advertisers in a Table)
⦁	Advertiser Details (Shows the details of the Advertiser record)
⦁	Campaigns (Shows the campaigns related to the Advertiser record)
⦁	Ad Groups (Shows the Ad Groups assigned to the related Campaign that is related to the Advertiser Record)
⦁	Sales Orders (Shows the Sales Orders related to the Advertiser Record)
⦁	Connectors (Shows the third-party connections for the Advertiser Record)
⦁	Panel (This will be a fly-in panel from the right side, that will be shown for Create/Edit and Export features)
Assets:
⦁	images
⦁	js
⦁	css

1.	Use separate HTML files for each Page.
2.	Use separate JS files for each JS Feature.
3.	Use Global CSS files, and a separate CSS file for each page.

Account Lookup: Secondary modal with live filtering; selecting fills account + name and locks fields.

Persistence: localStorage keys for data and filters; selected advertiser stored for navigation.

State: advertisers, filters, pagination, searchQuery, selectedAccount.
Methods: init, load/saveAdvertisers, cacheElements, initEventListeners, applyFiltersAndRender, renderTable, renderPagination, filtering helpers, modal controls, validation, duplicate detection.
Event Delegation: Table row clicks and notification list handling via parent listeners.
Defensive Checks: Guard for missing DOM nodes; try/catch around storage and initialization.
Styling / UI Conventions

Bootstrap utility-first classes; custom inline styles for a few brand colors (#004E8B) and minor overrides.
Components:
Header: Flex with grouped interactive icons.
Toolbar: Button group + filters + search + vertical separators (borders).
Table: Striped via divide utilities; compact row spacing (py-2).
Modals: Centered fixed overlay with rounded, shadowed panels.
Buttons: Consistent hover:bg-gray-50 or hover:bg-gray-200; primary actions use bg-blue-600.
Form Fields: Standard border-gray-300 + focus ring (Bootstrap focus classes).
Error Messages: Hidden by default; toggled via classList.
State & Naming

Data Keys: id, name, account, status, campaign, adGroup, owner.
Storage Keys: adv_list_data, adv_list_state, adv_list_owner, adv_selected_id, adv_selected_data, adv_draft.

Validation Rules

Account + Name required unless “No CRM Account” checked.
Duplicate checks: exact match on account; case-insensitive match on name.
Platforms required only if platform toggle enabled.
Create button enabled only when all active rules pass.
Extensibility Notes

Sorting UI present; add sort logic by attaching to .sort-btn buttons.
Consider extracting repeated DOM queries to cached refs (already partially done).
Potential enhancement: central event bus or splitting modules if page grows.
Style Guide (Derived)

Use Bootstrap utilities; avoid custom CSS unless necessary (tooltips, search animation).
Keep component state inside a single app object; avoid global loose variables.
Always null-check elements before attaching listeners.
Use localStorage for lightweight persistence; wrap in try/catch.
Prefer semantic, lowerCamelCase for JS properties; UPPER_SNAKE for config constants.
Show/hide via utility classes (hidden) not inline display changes.
Keep modals self-contained; open/close functions manage body scroll locking.
Add accessibility: aria-labels for icon-only buttons, role/label on alerts.

Extensibility Patterns

Add new panel types by:
Create getXForm() returning HTML string.
Add case in openPanel switch.
Handle save branch in panel-save-btn listener.
For validation: Inject data attributes (data-required) and run a validatePanel() before enabling Save.
Best Practices

Keep business logic (data persistence) separated from HTML string builders.
Debounce expensive DOM ops; currently panel content replaced once per open (fine).
Avoid inline alert() for production; replace with toast or inline validation messages.
Use semantic inputs (type="date","email","number") for built-in validation and mobile keyboards.
